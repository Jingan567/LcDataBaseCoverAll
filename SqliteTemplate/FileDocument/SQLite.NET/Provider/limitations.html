此 ADO.NET SQLite 数据提供程序的限制

随着提供商的发展，这个没有太多限制。SQLite 没有 支持行级或表级锁。当连接锁定数据库以进行写入时，在写入作完成之前，其他连接或进程都不能对数据库进行读取或写入作。SQLite.NET 提供程序尝试重试 在内部，如果数据库被锁定，则最高为 命令有问题。

SQLite 本质上是无类型的，并且只理解几种基本数据类型 本地。他们是 （在 .NET-speak） Int64、Double、String 和 Blob。这 SQLite.NET 提供程序将使用它可以收集到的数据库架构信息 强制类型化，但这是一门不精确的科学。

不支持分层 DataReader。在 的情况下，在连接上创建的任何 SQLiteCommand 都将（当 executed） 自动加入正在进行的事务，无论 该事务是在命令之前或之后创建的。

可以重新为 SQLiteCommand 对象分配新的 SQLiteConnection 对象 只要命令上没有 DataReader 处于活动状态。

打开事务被视为写入作，因此请仅在 您想要写入数据库！如果您按住 open 一个 “immediate” transaction 时，其他连接上的所有读取器都将被阻塞，直到 交易已关闭！

线程安全

SQLite 中的多线程必须小心完成。以下是限制：

  • 您可以Clone（） 一个 SQLiteConnection 对象，并将克隆的对象传递给 另一个线程。传递后，另一个线程将成为 cloned 连接，并且原始线程不得保留对 clone 或对 clone 调用任何方法。
  • 您可以创建多个线程，这些线程可以创建自己的线程 SQLiteConnection 和后续对象来访问数据库。 多个线程上指向同一数据库文件的多个连接是 完全可以接受，并且行为可预测。
  • 您不得调用方法或属性或以其他方式引用任何 SQLite 提供程序类 属于另一个线程。
  • 您不能传递 SQLiteCommand、SQLiteDataReader、SQLiteDataAdapter 或 任何其他 SQLite 提供程序类，但克隆的 SQLiteConnection 到另一个类除外 线。

再次了解 SQLite 没有细粒度的锁定机制。是的 因此您自己负责在多线程环境中处理 如果一个线程中存在长时间运行的查询，则可能发生的超时 阻止执行另一个线程中的查询。这些超时只会 如果一个线程正在尝试读取，而另一个线程正在尝试读取，则会出现此事件 写。首先获得锁的线程将是要执行的线程，而 其他线程将阻塞，直到 CommandTimeout 值过去，或者其他 螺纹完成。
