有关优化查询的提示

接下来的几段将尝试为您提供一些基本的规则 通常加快您的查询速度，尤其是 SQLite 的不利程度 受您在其他 SQL 中可能认为理所当然的 SQL 行为类型 供应商。它绝不是完整的优化指南。更多 有关优化查询的详细信息，请访问 sqlite.org。

交易的重要性

如果您在 SQLite 中插入数据而不先启动事务：DO NOT PASS GO！立即调用 BeginTransaction（），并以 Commit（） 结束！如果你觉得我在开玩笑，请再想一想。SQLite 的 A.C.I.D. 设计意味着 每次在事务外部插入任何数据时，都会显示一个隐式的 构造 transaction，进行 INSERT，并销毁 transaction。每 时间。如果您想知道为什么您的插件需要 100 倍 比你想象的要长，不要再看了。

准备好的语句

让我们快速浏览一下以下代码并评估其性能：

      using (SQLiteCommand mycommand = new SQLiteCommand(myconnection))
      {
        int n;
        
        for (n = 0; n < 100000; n ++)
        {
          mycommand.CommandText = String.Format("INSERT INTO [MyTable] ([MyId]) VALUES({0})", n + 1);
          mycommand.ExecuteNonQuery();
        }
      }

这段代码看起来很紧凑，但如果你认为它表现良好，那你就死定了 错。这是它的问题所在：

  • 我没有先开始交易！这个插页很慢！
  • CLR 隐式调用 “new” 100,000 次，因为我正在格式化 string 在循环中
  • 由于 SQLite 预编译了 SQL 语句，因此引擎正在构造和 解构 100,000 个 SQL 语句并分配/释放其内存
  • 所有这些建设和破坏都涉及大约 300,000 名本地人 托管互作调用比优化的插入

因此，让我们稍微重写一下该代码：

      using (SQLiteTransaction mytransaction = myconnection.BeginTransaction())
      {
        using (SQLiteCommand mycommand = new SQLiteCommand(myconnection))
        {
          SQLiteParameter myparam = new SQLiteParameter();
          int n;
        
          mycommand.CommandText = "INSERT INTO [MyTable] ([MyId]) VALUES(?)";
          mycommand.Parameters.Add(myparam);
          
          for (n = 0; n < 100000; n ++)
          {
            myparam.Value = n + 1;
            mycommand.ExecuteNonQuery();
          }
        }
        mytransaction.Commit();
      }

现在，这是任何数据库引擎的极快插入，而不是 只是 SQLite。SQL 语句准备一次 —— 在第一次调用 ExecuteNonQuery（） 的 ExecuteNonQuery（） 中。一旦准备好，就永远不需要重新评估。此外 我们在循环中不分配内存，并且只执行非常少量的 互作转换。用事务将整个 Thing 括起来，然后 这个插件的性能比原来的要快得多，以至于它 值得一阵海盗般的笑声。

每个值得一提的数据库引擎都使用准备好的语句。如果你是 不为此编码，您就没有编写优化的 SQL，这就是底部 线。
