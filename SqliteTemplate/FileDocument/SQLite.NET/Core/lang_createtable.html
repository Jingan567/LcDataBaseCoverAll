2. CREATE TABLE 命令

“CREATE TABLE”命令用于在 SQLite 中创建新表 数据库。CREATE TABLE 命令指定 新建表：

  • 新表的名称。

  • 在其中创建新表的数据库。表格可以是 在 main 数据库、临时数据库或任何附加的 数据库。

  • 表中每列的名称。

  • 表中每列的声明类型。

  • 表中每列的默认值或表达式。

  • 用于每列的默认排序规则序列。

  • （可选）表的 PRIMARY KEY。单列和 支持复合（多列）主键。

  • 每个表的一组 SQL 约束。SQLite 支持 UNIQUE，NOT NULL、CHECK 和 FOREIGN KEY 约束。

  • （可选）生成列约束。

  • 该表是否为 WITHOUT ROWID 表。

  • 表是否受到严格的类型检查。

每个 CREATE TABLE 语句都必须为新 table 指定一个名称。 以 “sqlite_” 开头的表名保留供内部使用。它 是尝试创建名称以 “sqlite_”。

如果指定了 schema-name，则它必须是 “main”， “temp” 或附加数据库的名称。在这种情况下 将在 named 数据库中创建新表。如果 “TEMP” 或 “TEMPORARY” keyword 出现在 “CREATE” 和 “TABLE” 之间，则新表为 在 Temp 数据库中创建。除非 schema-name 为 “temp”，否则同时指定 schema-name 和 TEMP 或 TEMPORARY 关键字是错误的。 如果未指定 schema 名称，并且 TEMP 关键字不存在，则会在主 数据库。

尝试在数据库中创建新表通常是错误的 already 包含同名的表、索引或视图。但是，如果 “IF NOT EXISTS” 子句指定为 CREATE TABLE 语句的一部分，并且 已存在同名的表或视图，则 CREATE TABLE 命令 根本没有效果（并且不会返回错误消息）。错误仍然是 如果由于现有索引而无法创建表，则返回，即使 如果指定了 “IF NOT EXISTS” 子句。

使用 DROP TABLE 语句删除表。

3. 列定义

除非它是一个 CREATE TABLE ...AS SELECT 语句中，CREATE TABLE 包括 一个或多个列定义，（可选）后跟表约束列表。每个列定义都包含 列的名称，（可选）后跟列的声明类型， 然后是一个或多个可选的 column constraints。包含在 “column constraints” 的定义，用于前面的 statement 是 COLLATE 和 DEFAULT 子句，即使它们并不是真正的 constraints 的含义是它们不限制 table 可以 包含。其他约束 - NOT NULL、CHECK、UNIQUE、PRIMARY KEY 和 FOREIGN KEY 约束 - 对表数据施加限制。

表中的列数受 SQLITE_MAX_COLUMN 编译时参数的限制。表的单行不能存储超过 SQLITE_MAX_LENGTH 字节的数据。这两个限制都可以降低到 使用 sqlite3_limit（） C/C++ 接口的运行时。

3.1. 列数据类型

与大多数 SQL 数据库不同，SQLite 不限制数据类型 可以根据 Columns declared type 插入到列中。相反 SQLite 使用动态类型。列的声明类型用于 仅确定列的关联性。

3.2. DEFAULT 子句

DEFAULT 子句指定要用于列的默认值，如果没有 value 由用户在执行 INSERT 时显式提供。如果有 没有显式的 DEFAULT 子句附加到列定义，则 该列的默认值为 NULL。显式 DEFAULT 子句可以指定 默认值为 NULL、字符串常量、blob 常量、 signed-number 或括在括号中的任何常量表达式。一个 default 值也可以是与大小写无关的特殊关键字之一 CURRENT_TIME、CURRENT_DATE 或 CURRENT_TIMESTAMP。就 DEFAULT 子句中，如果表达式 不包含子查询、列或表引用、绑定参数、 或用双引号而不是单引号括起来的字符串文本。

每次通过 INSERT 语句将行插入表中时， 不为存储在 新行由其默认值确定，如下所示：

  • 如果列的默认值为常量 NULL、text、blob 或 signed-number 值，则该值将直接在新行中使用。

  • 如果列的默认值是括号中的表达式，则 对于插入的每一行，表达式都会计算一次，结果会 在新行中使用。

  • 如果列的默认值为 CURRENT_TIME，则 CURRENT_DATE 或 CURRENT_TIMESTAMP，则新行中使用的值为文本 当前 UTC 日期和/或时间的表示形式。CURRENT_TIME， 格式为 “HH：MM：SS”。对于CURRENT_DATE，“YYYY-MM-DD”。这 CURRENT_TIMESTAMP格式为“YYYY-MM-DD HH：MM：SS”。

3.3. COLLATE 子句

COLLATE 子句指定要用作的整理序列的名称 列的默认排序规则序列。如果没有 COLLATE 子句为 指定，则默认排序规则序列为 BINARY。

3.4. GENERATED ALWAYS AS 子句

包含 GENERATED ALWAYS AS 子句的列是生成列。 从 SQLite 版本 3.31.0 （2020-01-22） 开始支持生成列。 有关功能和 生成列的限制。

3.5. 主键

SQLite 中的每个表最多只能有一个 PRIMARY KEY。如果 关键字 PRIMARY KEY 添加到列定义中，则主键 因为 table 由该单个列组成。或者，如果 PRIMARY KEY 子句 指定为 table-constraint，则 table 的主键 由指定为 PRIMARY KEY 子句一部分的列列表组成。 PRIMARY KEY 子句必须仅包含列名 — 使用 不支持 PRIMARY KEY 的索引列中的表达式。 如果 CREATE TABLE 语句。PRIMARY KEY 对于普通表是可选的 但对于 WITHOUT ROWID 表是必需的。

如果表具有单个列主键及其声明的类型 column 为 “INTEGER” 且表不是 WITHOUT ROWID 表， 则该列称为 INTEGER PRIMARY KEY。 有关特殊属性和行为的说明，请参见下文 与 INTEGER PRIMARY KEY 相关联。

表中具有主键的每一行都必须具有唯一的组合 of 值。为了确定 主键值的唯一性，NULL 值被视为不同于 所有其他值，包括其他 NULL。如果 INSERT 或 UPDATE 语句尝试修改表内容，以便有两行或更多行 具有相同的主键值，即约束冲突。

根据 SQL 标准，PRIMARY KEY 应始终表示 NOT NULL。 不幸的是，由于某些早期版本中的一个 bug，这不是 case 中。除非该列是 INTEGER PRIMARY KEY 或 该表是 WITHOUT ROWID 表或 STRICT 表 或列声明为 NOT NULL， SQLite 允许在 PRIMARY KEY 列中使用 NULL 值。SQLite 可以固定为 符合标准，但这样做可能会破坏旧式应用程序。 因此，决定仅记录 SQLite 允许在大多数 PRIMARY KEY 列中使用 NULL。

3.6. UNIQUE 约束

UNIQUE 约束类似于 PRIMARY KEY 约束，不同之处在于 单个表可以具有任意数量的 UNIQUE 约束。对于每个 UNIQUE 约束，则每行必须包含唯一的组合 的列中由 UNIQUE 约束标识的值。 对于 UNIQUE 约束，NULL 值 被视为与所有其他值（包括其他 NULL）不同。 与 PRIMARY KEY 一样，UNIQUE 表约束子句必须包含 only column names — 使用 不支持 UNIQUE 表约束的索引列中的表达式。

在大多数情况下，UNIQUE 和 PRIMARY KEY 约束是通过在数据库中创建唯一索引来实现的。 （WITHOUT ROWID 表上的 INTEGER PRIMARY KEY 和 PRIMARY KEY 除外。 因此，以下架构在逻辑上是等效的：

  1. 创建表 t1（a， b 唯一）;

  2. 创建表 t1（a， b 主键）;

  3. 创建表 t1（a， b）;
    在 t1（b） 上创建唯一索引 t1b;

3.7. CHECK 约束

CHECK 约束可以附加到列定义，或者 指定为 table constraint。在实践中，这没有区别。每 将新行插入表中或更新现有行的时间， 计算与每个 CHECK 约束关联的表达式，并且 cast 转换为 NUMERIC 值，其方式与 CAST 表达式相同。如果 result 为零（整数值 0 或实数值 0.0），则约束 发生冲突。如果 CHECK 表达式的计算结果为 NULL，或者 任何其他非零值，则不属于约束冲突。 CHECK 约束的表达式不能包含子查询。

CHECK 约束仅在写入表时进行验证，而不是在写入表时进行验证 它被阅读。此外，CHECK 约束的验证可以是 使用 “PRAGMA ignore_check_constraints=ON;” 暂时禁用 陈述。因此，查询可能会产生 违反 CHECK 约束。

3.8. NOT NULL 约束

NOT NULL 约束只能附加到列定义 未指定为 table constraint。毫不奇怪，NOT NULL constraint 指示关联的列不能包含 NULL 值。 尝试在插入新行时将列值设置为 NULL，或者 更新现有 API 会导致 constraint 冲突。非 NULL 在查询期间不会验证约束，因此列的查询可能会 即使列标记为 NOT NULL ，也会生成 NULL 值，如果 数据库文件已损坏。

4. 约束强制

在 INSERT 和 UPDATE 期间以及 PRAGMA integrity_check 和 PRAGMA quick_check 检查约束，有时也会检查 由 ALTER TABLE 提供。查询和 DELETE 语句通常不验证约束。 因此，如果数据库文件已损坏（可能由外部 程序直接更改数据库文件而不通过 SQLite 库）查询可能会返回违反约束的数据。 例如：

CREATE TABLE t1(x INT CHECK( x>3 ));
/* Insert a row with X less than 3 by directly writing into the
** database file using an external program */
PRAGMA integrity_check;  -- Reports row with x less than 3 as corrupt
INSERT INTO t1(x) VALUES(2);  -- Fails with SQLITE_CORRUPT
SELECT x FROM t1;  -- Returns an integer less than 3 in spite of the CHECK constraint

可以使用 PRAGMA ignore_check_constraints=ON; 语句。

4.1. 对约束冲突的响应

对约束冲突的响应由约束冲突解决算法确定。每 PRIMARY KEY、UNIQUE、NOT NULL 和 CHECK 约束具有默认冲突 resolution 算法。PRIMARY KEY、UNIQUE 和 NOT NULL 约束可以是 显式分配了另一个默认冲突解决算法 在他们的定义中包括 conflict-clause。 或者，如果约束定义不包含冲突子句，则 默认冲突解决算法为 ABORT。 CHECK 约束的冲突解决算法始终为 ABORT。 （仅出于历史兼容性，允许使用 table CHECK 约束 具有冲突解决子句，但这不起作用。 不同的 constraints 中的 同一表可能具有不同的默认冲突解决算法。看 标题为 ON CONFLICT 的部分以获取更多信息。

5. ROWID 和 INTEGER PRIMARY KEY

除 WITHOUT ROWID 表外，SQLite 表中的所有行 具有一个 64 位有符号整数键，用于唯一标识其表中的行。 这个整数通常是 称为 “rowid”。rowid 值可以使用特殊的 不区分大小写的名称 “rowid”、“oid” 或 “_rowid_” 代替列名。 如果表包含名为 “rowid”、“oid” 或 “_rowid_” 的用户定义列， 那么该名称始终引用显式声明的列，并且不能使用 检索整数 rowid 值。

rowid（以及 “oid” 和 “_rowid_”） 在 WITHOUT ROWID 表中被省略。 WITHOUT ROWID 表仅在 SQLite 版本 3.8.2 （2013-12-06） 及更高版本中可用。 缺少 WITHOUT ROWID 子句的表称为“rowid 表”。

rowid 表的数据存储为 B 树结构，其中包含 每个表行一个条目，使用 rowid 值作为键。这意味着 按 rowid 检索或排序记录很快。搜索具有 特定 rowid，或者对于 rowid 在指定范围内的所有记录，为 大约是通过指定任何其他 PRIMARY 进行的类似搜索的两倍 KEY 或索引值。

除了下面提到的一个例外，如果 rowid 表的主键包括 的单个列，并且该列的声明类型是 “INTEGER” upper 和 small case，则该列将成为 rowid 的别名。这样的 column 通常称为 “integer primary key”。A PRIMARY KEY列 仅当声明的类型名称正好为 “整数”。其他整数类型名称，如 “INT” 或 “BIGINT” 或 “SHORT INTEGER” 或 “UNSIGNED INTEGER” 会导致主键列的行为与普通 table 列，而不是作为 rowid 的

上面提到的例外是，如果带有 声明的类型 “INTEGER” 包含一个 “PRIMARY KEY DESC” 子句，它没有 成为 rowid 的别名，并且未分类为整数主键。 这个怪癖不是故意的。这是由于 SQLite 早期版本中的一个错误。 但是修复此错误可能会导致向后不兼容。 因此，原始行为已被保留（并记录），因为 odd 极端情况下的行为比兼容性中断要好得多。这意味着 以下三个 table 声明都会导致列 “x” 为 rowid 的别名（整数主键）：

  • CREATE TABLE t（x 整数主键 ASC， y， z）;
  • 创建表 t（x INTEGER， y， z， PRIMARY KEY（x ASC））;
  • 创建表 t（x INTEGER， y， z， PRIMARY KEY（x DESC））;

但以下声明不会导致 “x” 成为 rowid 中：

  • CREATE TABLE t（x 整数主键 DESC， y， z）;

Rowid 值可以使用同一 方式，使用内置别名之一 （“rowid”、“oid” 或 “_rowid_”） 或使用由整数创建的别名 primary key 的 key 中。同样，INSERT 语句可以提供一个值，用作 rowid 中插入的每一行。与普通的 SQLite 列不同，整数主 key 或 rowid 列必须包含整数值。整数主键或 rowid 列无法保存浮点值、字符串、BLOB 或 NULL。

如果 UPDATE 语句尝试设置整数、主键或 rowid 列 更改为 NULL 或 blob 值，或者更改为不能无损的字符串或实数值 转换为整数时，会出现 “datatype mismatch” 错误，并且语句 将中止。如果 INSERT 语句尝试插入 blob 值或字符串 或无法无损地将整数转换为 整数主键或 rowid 列，则会出现 “datatype mismatch” 错误，并且 语句中止。

如果 INSERT 语句尝试将 NULL 值插入 rowid 或 Integer 主键列，系统会选择一个整数值作为 rowid 自动。有关如何执行此作的详细说明将单独提供。

不允许外键约束的父键 使用 rowid。父键必须仅使用命名列。
