INDEXED BY 子句

1. INDEXED BY 的工作原理

INDEXED BY 短语强制 SQLite 查询计划程序使用 DELETE、SELECT 或 UPDATE 语句上的特定命名索引。 INDEXED BY 短语是 SQLite 扩展， 不能移植到其他 SQL 数据库引擎。

qualified-table-name 的

“INDEXED BY index-name” 短语指定 命名索引 ，以便查找上表中的值。 如果 index-name 不存在或无法使用 ，则 SQL 语句的准备将失败。 “NOT INDEXED” 子句指定在以下情况下不应使用索引 访问上表，包括隐含索引 create by UNIQUE 和 PRIMARY KEY 约束。但是，即使指定了 “NOT INDEXED” ，rowid 仍可用于查找条目。

一些 SQL 数据库引擎提供了非标准的 “提示” 机制，这些 可用于为查询优化器提供有关它应该使用哪些索引的线索 用于特定语句。SQLite 的 INDEXED BY 子句不是提示机制，不应如此使用。 INDEXED BY 子句不会向优化器提供有关哪个索引的提示 使用;它为 Optimizer 提供了要使用的索引的要求。 如果查询优化器无法使用 INDEXED BY 子句，则查询将失败并显示错误。

INDEXED BY 子句不用于优化 查询的性能。INDEXED BY 子句的意图是 在架构更改（如 drop-ing或 创建索引，会导致时间敏感型查询的查询计划 进行更改。INDEXED BY 子句旨在帮助检测 回归测试期间不需要的查询计划更改。 应用 告诫开发人员在 应用程序设计、实施、测试和调整。如果 INDEXED BY 的 INDEX BY 是完全可以使用的，它应该插入到最 在“锁定”设计时结束开发过程。

2. 另请参阅

  1. 查询计划程序清单描述了应用程序 开发人员应遵循以帮助解决 Query Planner 问题。 请注意，使用 INDEXED BY 是最后的手段，只能使用 当所有其他措施都失败时。

  2. 一元 “+” 运算符可用于取消 WHERE 子句中的术语被索引使用的条件。 谨慎使用一元 + 有时有助于防止查询计划程序 选择一个较差的索引，而不限制它使用一个特定索引。 仔细放置一元 + 运算符是更好的控制方法 查询使用哪些索引。

  3. sqlite3_stmt_status（） C/C++ 接口与 SQLITE_STMTSTATUS_FULLSCAN_STEP 和 SQLITE_STMTSTATUS_SORT 动词一起 可用于在运行时检测 SQL 语句何时不是 有效利用指数。许多应用程序可能更愿意 使用 sqlite3_stmt_status（） 接口检测索引滥用 而不是此处描述的 INDEXED BY 短语。
